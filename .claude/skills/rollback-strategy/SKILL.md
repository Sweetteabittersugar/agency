# 回滚策略

## 概述
提供部署和发布的风险控制策略，覆盖数据库回滚、服务回滚、灰度发布、蓝绿部署等多种场景下的回滚决策和执行方案。

## 使用场景
- 上线前制定部署方案和回滚预案
- 线上事故需要紧急回滚
- 数据库迁移的风险评估和回滚验证
- 选择发布策略（滚动、蓝绿、金丝雀）

## 核心原则
1. **每次部署都必须有回滚方案**：没有回滚方案的部署就是赌博。回滚脚本必须经过预演验证。
2. **向前回滚优先于向后回滚**：首选"部署修复版本"而非"回退代码"。回退可能引入新的不一致（如 Schema 已变更）。
3. **数据库迁移是回滚最高风险点**：DDL 变更（删除列、修改类型）可能无法无损回滚。删除操作先标记 deprecated 再清数据。
4. **回滚决策矩阵**：P0 事故（数据丢失/完全不可用）→ 立即回滚或紧急修复；P1（功能降级）→ 评估回滚 vs 向前修复；P2-P3 → 修复优先。
5. **回滚要快**：回滚操作应在 5 分钟内完成。复杂的回滚流程说明部署架构设计有问题。

## 发布策略
- **蓝绿部署**：两套完整环境切换，回滚就是切回旧环境。零停机但资源翻倍。
- **金丝雀发布**：小比例流量先验证新版本，逐步扩大。问题影响面可控。
- **滚动更新**：逐实例替换，回滚是反向滚动。适合无状态服务。
- **功能标志（Feature Flag）**：代码已部署但通过开关控制。回滚就是关开关，最快最安全。

## 检查清单
- [ ] 每次部署是否有配套的回滚方案并经过预演？
- [ ] 数据库迁移是否有可执行且已验证的回滚脚本？
- [ ] 是否有决策矩阵指导不同严重级别的事故响应？
- [ ] 是否有功能标志机制支持代码级快速回滚？
- [ ] 回滚操作的平均执行时间是否在 SLA 范围内（建议 ≤5 分钟）？
