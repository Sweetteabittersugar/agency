# 架构设计模式

## 概述
提供常见软件架构模式的系统化参考，覆盖分层架构、微服务、事件驱动、CQRS 等主流风格，帮助在项目早期做出合理的架构决策。

## 使用场景
- 新项目启动阶段的技术选型
- 现有系统架构评审与重构规划
- 跨团队架构方案讨论的对齐参考
- 架构决策记录（ADR）的撰写依据

## 核心原则
1. **简单优先**：能用单体就不用微服务，能用单数据库就不用分布式。复杂度增加必须有对应的业务收益。
2. **先定义边界再选模式**：架构模式服务于业务领域边界，而不是反过来削足适履。
3. **非功能需求驱动**：可扩展性、可靠性、性能、安全性等非功能需求才是选型的核心依据。
4. **演进式设计**：架构不需要在第一天完美——预留扩展点，允许通过迭代演进。
5. **记录决策理由**：每个架构决策都要写 ADR，记录当时的选择、备选方案和权衡理由。

## 常见模式速查
- **分层架构**：展示层 → 业务层 → 持久层 → 数据层，适合中小型项目
- **六边形架构（端口适配器）**：领域核心通过端口与外部解耦，适合复杂业务
- **CQRS**：读写分离，适合读写负载差异大的场景
- **事件驱动**：通过事件总线解耦服务，适合异步流程和高吞吐
- **微内核**：核心 + 插件，适合产品平台化
- **微服务**：独立部署的服务单元，适合大规模团队和系统

## 检查清单
- [ ] 是否能用更简单的架构满足需求？
- [ ] 架构选型是否由明确的非功能需求驱动？
- [ ] 模块/服务边界是否与业务领域对齐？
- [ ] 是否有 ADR 记录关键决策及其理由？
- [ ] 是否预留了合理的演进路径和扩展点？
