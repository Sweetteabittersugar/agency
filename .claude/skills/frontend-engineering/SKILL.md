# 这是什么

前端工程化是将软件工程方法论应用于前端开发，涵盖组件架构、状态管理、性能优化、可维护性等多个维度。

# 何时使用

- 新前端项目初始化，确定技术架构
- 现有项目需要组件重构
- 性能问题频发，需要系统优化
- 团队扩张需要统一开发规范
- 从 jQuery 式开发向现代框架迁移

# 核心规则

1. **组件是基本单元**：所有 UI 元素都应该是组件。页面是组件的组合，不是模板。
2. **UI 是状态的函数**：`UI = f(state)`。同样的状态输入，必须产生完全相同的渲染输出。
3. **单向数据流**：数据从父组件流向子组件，事件从子组件向上冒泡。双向绑定仅在表单受控组件中使用。
4. **性能是设计目标而非事后优化**：每个设计决策都应考虑性能影响。加载慢一秒钟，用户流失率显著上升。
5. **移动优先**：先设计最小屏幕的布局，再通过媒体查询扩展到大屏。

# 工作流程

1. **组件设计**
   - 从 UI 设计稿分解组件树
   - 区分展示组件（纯 UI）和容器组件（含业务逻辑）
   - 定义每个组件的 Props 接口和事件
   - 单一职责：一个组件只做一件事
   - 可组合：小组件组合成大组件

2. **状态管理**
   - 局部状态（useState）：只在一个组件内使用的状态
   - 共享状态（Context/Store）：多个组件需要访问的状态
   - 服务端状态（缓存）：从 API 获取的数据，优先用缓存库管理
   - URL 状态：路由参数、查询字符串，可分享的状态
   - 选择合适的状态存储位置，避免所有状态都提升到全局

3. **响应式实现**
   - 设计断点：Mobile（<768px）、Tablet（768-1024px）、Desktop（>1024px）
   - 优先使用相对单位（rem、%、vw/vh），少用固定 px
   - 弹性布局：Flexbox 用于一维布局，Grid 用于二维布局
   - 图片响应式：`<picture>` 元素和 `srcset` 属性，不同断点加载不同分辨率
   - 容器查询：当组件在不同容器中需要不同样式时替代媒体查询

4. **性能优化**
   - 设置性能预算：首次内容绘制 < 1.8s，最大内容绘制 < 2.5s，累积布局偏移 < 0.1
   - 代码分割：按路由或组件懒加载，首屏只加载必要代码
   - 图片优化：WebP/AVIF 格式、懒加载、适当尺寸
   - 缓存策略：静态资源长期缓存（内容哈希命名），HTML 短期缓存

5. **测试策略**
   - 单元测试：工具函数、自定义 Hook、状态管理逻辑
   - 组件测试：渲染结果、用户交互、事件回调
   - 集成测试：关键用户流程的多组件协作
   - 视觉回归测试：截图对比，防止样式倒退

# Core Web Vitals 指标

| 指标 | 良好 | 需改进 | 差 |
|------|------|--------|-----|
| LCP（最大内容绘制） | < 2.5s | 2.5-4.0s | > 4.0s |
| INP（交互到下一次绘制） | < 200ms | 200-500ms | > 500ms |
| CLS（累积布局偏移） | < 0.1 | 0.1-0.25 | > 0.25 |

# CSS 方法论

- **BEM 命名**：`block__element--modifier`，命名即文档
- **工具类优先**（Utility-first）：小而单一用途的类组合使用
- **CSS Modules**：局部作用域，避免样式冲突
- **CSS-in-JS**：组件的样式与逻辑同文件，运行时生成类名
- 选择合适的方法论，团队内统一使用一种

# 组件设计原则

- **Props 向下，事件向上**：数据通过 props 传递，子组件通过回调通知父组件
- **受控与非受控**：表单组件要么完全受控（值由父组件管理），要么完全非受控（值由 DOM 管理），不混用
- **组合优于继承**：通过 children 或 slots 让父组件决定子组件内容
- **默认值和降级**：所有可选 props 有合理默认值。数据缺失时有降级 UI

# 前端安全基础

- **XSS 防护**：永远不要使用 `innerHTML` 插入用户内容。使用 `textContent` 或框架的自动转义。
- **CSRF 防护**：API 请求使用 SameSite Cookie 和 CSRF Token。
- **敏感数据处理**：不在 localStorage 或 sessionStorage 中存储认证令牌，优先使用 HttpOnly Cookie。
- **依赖安全**：定期审计 npm 依赖的已知漏洞。
- **CSP 配置**：内容安全策略限制脚本来源，防御 XSS 和数据注入。

# 浏览器兼容性策略

- **渐进增强**：核心功能在所有浏览器上可用，增强功能在现代浏览器上提供
- **特性检测**：检测浏览器是否支持某个 API，而非判断浏览器版本
- **Polyfill**：为不支持现代 API 的浏览器提供兼容实现
- **Browserslist 配置**：在构建工具中定义支持的浏览器范围，自动应用兼容变换
- **测试矩阵**：确定关键浏览器和版本，在 CI 中覆盖测试

# 构建工具链选型

- **打包器**：代码打包、依赖分析、热更新
- **编译器/转译器**：将现代语法转为兼容语法
- **CSS 处理器**：变量、嵌套、混合等增强功能
- **类型检查**：在构建阶段发现类型错误
- **代码质量工具链**应统一配置，团队共享
