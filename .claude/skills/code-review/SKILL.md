# 这是什么

代码审查是软件开发中通过同行检查发现缺陷、分享知识、确保代码质量的系统性实践。本 Skill 提供标准化的四维审查框架。

# 何时使用

- 功能分支合并到主分支之前
- 新人提交的代码需要 mentor 把关
- 对关键模块的修改需要二次确认
- 重构完成后验证行为不变
- 安全敏感功能需要额外审查

# 核心规则

1. **先看整体再扣细节**：先理解改动的目的和架构，再检查命名、格式等问题。不要在第一次浏览时就陷入变量命名争论。
2. **以行为为准而非作者**：评论针对代码本身，不针对编写的人。"这段逻辑在空数组时会越界"而非"你这里写错了"。
3. **区分阻塞和非阻塞**：安全漏洞、逻辑错误、数据丢失风险是阻塞项。风格偏好、命名建议是非阻塞项。
4. **每次审查不超过 400 行**：超过的改动要求拆分成更小的提交。审查效率随规模急剧下降。
5. **提供具体建议而非笼统批评**："这个函数太长了"不如"建议将 45-78 行的校验逻辑提取为独立函数"。

# 工作流程

1. **概览阶段**
   - 阅读提交信息和 PR 描述，理解改动的业务目的
   - 检查文件列表，评估改动范围和影响面
   - 确认是否有意外的文件被修改（如配置文件、二进制文件）

2. **正确性审查**
   - 逻辑是否按预期工作？考虑正常路径和异常路径
   - 边界条件是否处理？空值、零值、溢出、并发竞态
   - 是否有现成库可用而重复造轮子？
   - 类型是否正确？转换是否有精度损失？
   - 日期时区、货币精度、浮点比较是否正确？

3. **简洁性审查**
   - 是否存在死代码、注释掉的代码、调试日志？
   - 是否有不必要的抽象层或过度设计？
   - 变量和函数命名是否清晰表达意图？
   - 注释是否解释"为什么"而非"做什么"？

4. **可维护性审查**
   - 函数是否足够短小（建议不超过 50 行）？
   - 文件是否聚焦单一职责（建议不超过 800 行）？
   - 测试覆盖是否充分？关键路径是否都有测试？
   - 有没有硬编码的值应提取为配置？

5. **安全性审查**
   - 用户输入是否做了校验和清理？
   - 是否存在注入风险（SQL、命令、XSS）？
   - 认证和授权是否正确实施？
   - 是否有密钥、密码、Token 硬编码？
   - 日志中是否可能泄露敏感信息？

6. **输出评级**
   - **PASS**：代码质量符合标准，无阻塞问题，可以直接合并
   - **NEEDS_WORK**：存在需要修改的问题，但不影响核心逻辑。修改后可合并
   - **BLOCKED**：存在安全漏洞、逻辑错误或数据完整性风险，必须修改后重新审查

# 审查优先级

| 优先级 | 关注点 | 必须修复 |
|--------|--------|----------|
| P0 | 安全漏洞、数据丢失、崩溃 | 是（阻塞） |
| P1 | 逻辑错误、边界条件遗漏 | 是（阻塞） |
| P2 | 性能退化、资源泄漏 | 是 |
| P3 | 可读性差、命名不当 | 建议 |
| P4 | 代码风格、格式 | 建议（自动化工具处理） |

# 友善评论规范

- 使用询问语气而非命令："有没有考虑这里用 `filter` 替代 `for` 循环？"
- 当指出问题时同时给出改进建议，降低修复门槛
- 对好的实现给予肯定，正面反馈和建设性批评同样重要
- 如果离线审查发现有大量问题，考虑同步沟通而非堆积评论

# 自动化辅助

- 格式检查（Prettier、Black）应自动化，无需人工审查
- 静态分析（ESLint、Pylint、Clippy）结果应在审查前修复
- 类型检查应通过 CI 门禁
- 测试覆盖率报告应作为审查参考

# 参考标准

- Google 代码审查最佳实践（Google Engineering Practices）
- SmartBear《Best Kept Secrets of Peer Code Review》
- Conventional Comments 格式（`suggestion:`, `issue:`, `praise:` 前缀）
- 审查速度建议：每小时不超过 500 行
- 审查频率：单个审查会话不超过 60 分钟
- 审查者选择：至少一人熟悉该模块，一人不熟悉（带来新鲜视角）
- 审查周转时间：小型改动应在 4 小时内完成审查，中型改动不超过 1 个工作日
