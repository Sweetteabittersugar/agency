# 这是什么

CI/CD 是持续集成和持续交付/部署的工程实践。通过自动化流水线将代码从提交到生产的过程标准化，减少人工干预，提高交付速度和可靠性。

# 何时使用

- 新项目搭建自动化流水线
- 现有流水线优化和加速
- 从手动部署向自动化部署迁移
- 多环境部署策略设计
- 流水线故障排查和可靠性提升

# 核心规则

1. **主干开发，频繁集成**：每天至少合并一次到主干。分支存活超过一天就开始积累集成风险。
2. **流水线即代码**：CI/CD 配置存储在版本控制中，与代码一同管理。手工配置的控制台操作不可追溯、不可复现。
3. **失败即停止**：任何阶段失败，后续阶段不执行。不让有问题的代码继续往下走。
4. **不可变制品**：构建产物只建一次，在各环境间提升，而非每个环境重新构建。不同构建之间可能引入微妙差异。
5. **部署与发布分离**：部署是把代码放到生产环境，发布是让用户看到新功能。通过功能开关，可以先部署再择机发布。

# 工作流程

1. **触发阶段**
   - 代码推送到仓库触发流水线
   - Pull Request 触发轻量检查（lint、单元测试）
   - 合并到主干触发完整流水线
   - 支持手动触发用于特殊场景

2. **构建阶段**
   - 拉取代码，安装依赖
   - 代码静态分析和安全检查
   - 编译/打包（如需要）
   - 生成构建产物（JAR、Docker 镜像、静态文件）
   - 为产物打上唯一版本标签

3. **测试阶段**
   - 单元测试：每个模块的独立测试
   - 集成测试：模块间交互测试
   - 安全扫描：依赖漏洞、密钥检测、SAST
   - 并行执行无依赖的测试套件以缩短时间
   - 生成测试报告和覆盖率报告

4. **部署阶段**
   - 将构建产物部署到目标环境
   - 执行数据库迁移（应向后兼容）
   - 健康检查确认服务正常启动
   - 冒烟测试验证核心功能可用

5. **发布阶段**
   - 灰度发布：先部署到少量节点观察
   - 监控关键指标（错误率、延迟、业务指标）
   - 异常时自动回滚
   - 确认稳定后全量发布

# 环境策略

| 环境 | 用途 | 部署方式 | 数据 |
|------|------|----------|------|
| Dev | 开发自测 | 手动/自动 | 模拟数据 |
| Test/QA | 测试验证 | 每次提交自动 | 测试数据 |
| Staging | 预发布验证 | 通过测试后自动 | 脱敏生产数据 |
| Production | 用户服务 | 人工确认后自动 | 真实数据 |

- 各环境隔离，配置通过环境变量注入
- Staging 应尽量接近生产配置
- 生产环境部署前必须有人工确认环节

# 分支策略

- **Trunk-based**：所有开发在主干或极短分支（不超过一天）上进行。通过功能开关控制未完成功能。
- **GitHub Flow**：从主干创建功能分支，通过 PR 合并回主干，合并后自动部署。
- **GitFlow**：适合有明确发布周期的项目。有 develop、release、hotfix 等多条长期分支。

选择原则：发布频率越高，分支模型越简单。

# 回滚机制

- **数据库向前兼容**：迁移脚本应该是可重复执行且不回滚的。回滚用新的迁移脚本而非反向操作。
- **蓝绿部署**：维护两套完全相同的生产环境，部署到非活跃环境，验证后切换流量。
- **金丝雀部署**：逐步将流量从旧版本切换到新版本，持续监控。
- **制品版本化**：每个构建产物标签明确，可以一键回滚到任意历史版本。

# 参考标准

- Jez Humble & David Farley《持续交付》—— CI/CD 领域经典
- DORA 指标：部署频率、变更前置时间、变更失败率、故障恢复时间
- GitHub Actions / GitLab CI / Jenkins —— 主流流水线平台
- CNCF 云原生 CI/CD 白皮书
- Trunk Based Development 官方指南
- 流水线执行时间目标：从提交到部署不超过 10 分钟（快速反馈），最长不超过 30 分钟
- 并行化原则：无依赖的阶段并行执行，有依赖的阶段串行执行
