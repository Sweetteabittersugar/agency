# 这是什么

管线门控是一种结构化的质量保障模式。它将软件开发过程划分为多个阶段，每个阶段结束时设一道「门」，满足准入条件才能进入下一阶段，不满足则回退修复。

# 何时使用

- 多阶段复杂任务，如功能开发、系统重构
- Agent 系统中的任务编排
- 需要多人协作的大项目
- CI/CD 流水线的检查点设计
- 要求严格质量控制的项目管理

# 核心规则

1. **每道门必须可自动化验证**：准入准出条件不能依赖主观判断。必须能通过脚本、测试或检查工具自动得出「通过」或「不通过」。
2. **失败不跳过**：任何一道门失败，后续阶段不得启动。只修当前阶段的问题，不暗中修复下游问题的漏洞。
3. **门控配置即代码**：每道门的规则存储在版本库中，随项目演进同步更新。不在人的脑子里。
4. **门控要轻量**：检查项应该是必要的，不是越多越好。每增加一道检查，评估其与价值的权衡。
5. **失败信息要可操作**：门控失败时，输出必须指明具体问题、位置和修复建议。

# 工作流程

整个管线分为四个阶段，每阶段结束有一道门：

## 阶段 1：Plan（规划）

- **入口条件**：需求描述已明确，不含歧义
- **输出**：实施计划，含任务分解、依赖关系、风险评估
- **门控 1 — 计划审查**
  - 准出条件：任务分解粒度合理（单个任务不超过半天工作量）；依赖关系无循环；风险已识别并有缓解方案
  - 失败处理：返回细化需求或重新分析

## 阶段 2：Implement（实施）

- **入口条件**：通过门控 1
- **输出**：代码实现、相关文件变更
- **门控 2 — 实现审查**
  - 准出条件：所有新增代码已在对应文件中；无调试代码残留；代码风格符合项目规范；无硬编码敏感信息
  - 失败处理：返回修改代码后重新提交审查

## 阶段 3：Review（审查）

- **入口条件**：通过门控 2
- **输出**：审查报告，含评级和修改建议
- **门控 3 — 审查门控**
  - 准出条件：无阻塞级安全或逻辑问题；所有 P0/P1 问题已修复；审查评级为 PASS
  - 失败处理：NEEDS_WORK 要求修改后重新审查；BLOCKED 要求修复后重新审查

## 阶段 4：Verify（验证）

- **入口条件**：通过门控 3
- **输出**：测试报告、验证结果
- **门控 4 — 验证门控**
  - 准出条件：所有测试通过；覆盖率不低于基线；无回归问题；构建成功
  - 失败处理：返回修复失败项后重新验证

# 门控配置模板

每道门的配置包含以下字段：

- **门名称**：唯一标识
- **检查项列表**：每项包含检查类型、检查逻辑、失败级别（阻塞/警告）
- **执行时机**：自动触发还是手动触发
- **超时时间**：单个检查项的超时
- **重试策略**：失败后的重试次数和间隔
- **上报规则**：失败时通知谁

# 失败处理协议

- **阻塞级失败**：立即停止管线，通知相关人员，不允许继续
- **警告级失败**：记录下来，允许继续，但累积 N 个警告自动升级为阻塞
- **异常失败**（检查工具自身出错）：区分是工具错误还是代码问题。工具错误不应阻塞管线，但应告警
- **超时失败**：单检查项超时后标记为跳过并告警，不阻塞管线。连续超时升级为阻塞

# 管线可视化

建议维护一个门控状态面板，显示：
- 当前阶段和进度条
- 每道门的状态（通过 / 进行中 / 失败 / 未开始）
- 失败项的详细信息
- 预计完成时间

# 参考标准

- Jenkins Pipeline Stage/Gate 模式
- GitLab Merge Request Approvals 机制
- Spinnaker 部署门控策略
- 质量控制中的「检验门」概念（源自制造业的统计过程控制）
- 软件工程中的阶段门模型（Phase-Gate Model）
- 门控指标监控：记录每道门的通过率、失败原因分布、平均修复时间，持续改进门控规则
- 门控不应成为瓶颈：单道门检查时间不超过 10 分钟，否则考虑拆分或异步化
